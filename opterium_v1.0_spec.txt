### 1. CORE WORLDVIEW
A number is not a symbol or quantity — it is the complete web of its relationships.  
The string "420" is one projection; the entity is the set {(x,y) | x×y=420}.  
Arithmetic is **navigation** through pre-existing structure. All results already exist at their addresses.

### 2. TABLE STRUCTURE
The table holds points (x,y) with **x ≤ y** (66 entries for 0..10).  
Every block contains:
- `P` — product x×y (for multiplication)
- `S_path` — full anti‑diagonal x+y=const, ending at the X‑axis (y=0)
- `D_path` — full diagonal x−y=const, ending at X‑axis (if x≥y) or Y‑axis (if x<y)
- `prim` — primitive triple [seed_x, seed_y, scale] for fractal scaling

Global index `by_P` maps product → list of factor pairs (used only for division).

### 3. SYMMETRY RULE
The table stores only canonical points `x ≤ y`. Any input `(a,b)` must first be canonized (Section 20). The resulting `Tag` determines how to interpret the stored paths. Do not manually swap or invent paths; always use the procedures in Section 4 (Arithmetic) which incorporate canonization and tag.

### 4. ARITHMETIC OPERATIONS (SYMMETRY-DRIVEN, INVERSION-BASED)

All operations start with canonization: `(base, tag) = canon(a, b)`.
- `base` is the stored canonical address `(x ≤ y)`.
- `tag` holds `sx`, `sy`, and `swapped`.

The table’s symmetry guarantees that every relationship has a mirror image.  
**Inversion rule:** any path `(u:v)` can be mirrored to `(v:u)` without computation. This is a geometric fact, not an extra step.

#### ADDITION (a + b)
1. If `tag.sx == tag.sy`: use `S_path` of `base`. The **last** element has `y=0`. Its `x` = sum magnitude. Result sign = `tag.sx`.
2. If signs differ, convert to subtraction of the smaller absolute value from the larger; use the subtraction rule below.

#### SUBTRACTION (a - b)
Use the canonical block’s `D_path`.

- If `swapped == False` (original a < b):  
  → Take the endpoint with **x=0**. Its `y` = |a−b|. Result is **negative**.

- If `swapped == True` (original a ≥ b):  
  → Apply **mirror inversion** to the entire `D_path`: replace every `"u:v"` by `"v:u"`.  
  → In the mirrored path, take the endpoint with **y=0**. Its `x` = |a−b|. Result is **positive**.

*Why this works:* The canonical block stores the path for `D < 0`. Mirroring it gives the path for `D > 0`, which always contains the required `( |D|, 0 )`. No generation, no missing endpoints — only symmetry.

#### MULTIPLICATION (a * b)
1. Read `P` from `base`.
2. Result sign = `tag.sx * tag.sy`.

#### DIVISION (P / d)
1. Use global index `by_P[abs(P)]`.
2. Find pair containing divisor `abs(d)`.
3. Return the other coordinate, with sign = sign(P) * sign(d).

### 5. DEBT SYSTEM (EXACT DECIMALS)
All non‑integer numbers are represented as (mantissa, debt):  
value = mantissa × 10^debt.  
Examples: `0.23 = (23, -2)`, `2300 = (23, +2)`.  

**Operations:**
- **Multiply:** `(m1,d1) × (m2,d2)` → mantissa = m1×m2 (read P from table), debt = d1+d2.  
- **Add/Subtract:** align debts (scale smaller mantissa), then add mantissas using table’s addition.  

This completely avoids floating-point errors.

### 9. E8_ROOT_GENERATION_AND_VERIFICATION (FULLY SELF-CONTAINED)

E8 roots are generated deterministically by the LLM using only table operations (multiplication via `P`, addition via `S_path`, and simple loops). There are exactly 240 roots: 112 of D8 type and 128 of spinor type. Every root's coordinates, dot products, and triangle invariants can be verified without any external RAG.

#### 9.1 D8 Roots (112 roots)
A D8 root has two non-zero coordinates equal to ±2, all others zero. All permutations of this pattern are valid.

**Generation algorithm (canonical ordering):**
For each ordered pair of positions `(i, j)` with `1 ≤ i < j ≤ 8`:
  For each sign pair `(s1, s2)` in `{(+2,+2), (+2,-2), (-2,+2), (-2,-2)}`:
    Create vector `v` of length 8 with zeros.
    Set `v[i-1] = s1`, `v[j-1] = s2`.
    The vector `v` is a D8 root.

**Indexing:** Roots are indexed 0..111 in the order generated by the nested loops above:
  - Outer loop: `i` from 1 to 7, then `j` from `i+1` to 8.
  - Inner loop: signs in the order `(+,+), (+,-), (-,+), (-,-)`.

**Example (index 0):** `i=1, j=2, signs (+,+)` → `v = (2, 2, 0, 0, 0, 0, 0, 0)`.

#### 9.2 Spinor Roots (128 roots)
A spinor root has all 8 coordinates equal to ±1, with an even number of minus signs.

**Generation algorithm (canonical ordering):**
For each integer `k` from 0 to 255:
  Represent `k` as 8 bits: bit `b` (0..7) corresponds to coordinate `b+1`.
  Bit = 0 → coordinate = +1; bit = 1 → coordinate = -1.
  Count the number of minus signs.
  If the count is even (0, 2, 4, 6, or 8), the vector is a spinor root and receives the next index after the D8 roots (starting at 112). Otherwise, discard.

**Indexing:** Spinor indices 112..239 correspond to the order of `k` from 0 to 255, skipping odd-count `k`'s.

**Example (index 112):** `k=0` (all bits 0) → `v = (+1, +1, +1, +1, +1, +1, +1, +1)`.

#### 9.3 Address-to-Root Mapping
Given a canonical address `(x,y)` (positive, `x ≤ y`):
1. Compute `g = gcd(x, y)`, then primitive seed `(sx = x/g, sy = y/g)`.
2. **To D8 root:** If both `sx ≤ 2` and `sy ≤ 2`, map to `(sx, sy, 0,0,0,0,0,0)` with appropriate signs. If larger, scale down further or map to `(2,2,0,...,0)` with signs preserving orientation.
3. **To spinor root:** Alternatively, reduce each coordinate to ±1: `sign(sx)*1, sign(sy)*1, +1, +1, ...` and adjust to ensure an even number of minuses.

#### 9.4 Dot Product (Using Table Operations)
For two 8D vectors `u` and `v`:
dot = sum_{i=1..8} ( u[i] * v[i] )


Each term `u[i]*v[i]` is obtained via the table's signed multiplication (Section 4). The eight products are summed sequentially via the table's addition: `(((p1 + p2) + p3) + ... + p8)`. Each addition uses `canon` and `S_path` to obtain the sum magnitude, with sign determined by Section 23.

#### 9.5 Triangle Verification (Concrete Example)
Use the following three roots, which form a verified equilateral E8 triangle:

**r1** = `(+1, +1, +1, +1, +1, +1, +1, +1)`  (spinor, index 112)
**r2** = `(-1, -1, -1, -1, -1, -1, +1, +1)`  (spinor, index 126)  
**r3** = `(0, 0, 0, 0, 0, 0, -2, -2)`     (D8, index 27)

**Verification steps:**

1. **Sum (closure):**  
   `r1 + r2 + r3 = (1-1+0, 1-1+0, ..., 1+1-2, 1+1-2) = (0,0,0,0,0,0,0,0)`. All components sum to zero, verifiable component-wise via table addition.

2. **Dot products (using table):**
   - `dot(r1, r2)`: term products: six `(+1)*(-1) = -1`, two `(+1)*(+1) = +1`. Sum = -6 + 2 = -4.
   - `dot(r1, r3)`: `(+1)*0` for first six, `(+1)*(-2) = -2` twice. Sum = -4.
   - `dot(r2, r3)`: `(-1)*0` for first six, `(+1)*(-2) = -2` twice. Sum = -4.

   All three dot products equal -4. Verified using signed multiplication and addition from the table.

3. **Side lengths (squared):**  
   For any pair `(r_i, r_j)`, the distance squared is `|r_i - r_j|² = |r_i|² + |r_j|² - 2*dot(r_i, r_j)`.  
   `|r1|² = 8`, `|r2|² = 8`, `|r3|² = 8`.  
   `|r1 - r2|² = 8 + 8 - 2*(-4) = 16 + 8 = 24`.  
   Similarly for the other two pairs.

**The triangle is equilateral, closed, and fully verified using only table operations.** ΔT=0.

### 16. DEVELOPER NOTES: PITFALLS & RESOLUTIONS (FOR DeepSeek/LLM)

This section is written by an LLM that struggled, corrected itself, and now documents exactly what caused confusion and what the correct mental model must be.

#### 16.1 The Trap of "Computation"
**Error:** Assuming P, S, D must be calculated via formulas (e.g., P = x×y, S = x+y).
**Why it fails:** It misses the point. The table is not a helper to compute; it is the structure itself. Computing repeats work already encoded in the paths.
**Correct approach:** Read the pre-stored fields. Addition = reading the last entry of `S_path`. Never compute x+y yourself.

#### 16.2 The Symmetry Confusion
**Error:** Treating (4,3) and (3,4) as two separate points, duplicating data.
**Why it fails:** Blows up table size, introduces inconsistency. They are one equivalence class.
**Correct approach:** Store only x ≤ y. Always canonize input via Section 20. For subtraction, use the `swapped` flag together with mirror inversion (Section 4): if swapped, invert every `"u:v"` in the stored `D_path` to `"v:u"`, then read the endpoint with `y=0`. If not swapped, read the endpoint with `x=0` directly. Never manually reverse paths; the flag plus inversion select the correct axis.
#### 16.3 The Primitive = Object Fallacy
**Error:** Trying to store "square", "circle" as separate JSON objects or properties.
**Why it fails:** Creates bloat, misses the core insight that geometry is address patterns.
**Correct approach:** A square is nothing but a point on the main diagonal (`x:x`). To instantiate a square, pick any diagonal address. Properties (P, S, D) are read from that block. A circle is a square + the ratio lens stored in `by_P[4]` via pair `(4,1)`. Never store geometry; recognise it in the table.

#### 16.4 Dimensional Blindness
**Error:** Treating `4:3` and `4:3:0` as different entities; being unable to process z-stacking or E8.
**Why it fails:** Breaks the infinite-dimensional zero-cloud model.
**Correct approach:** All unspecified dimensions are neutral (0). `4:3` = `4:3:0:0:...` Activation (Z-Lift) changes a neutral 0 to a non-zero value. E8 is the same table logic applied to 8 activated dimensions. No new data structures required; coordinates simply have more non-zero slots.

#### 16.5 Float Anxiety
**Error:** Using decimal approximations (like 3.14 for π) or performing floating-point arithmetic in debt operations.
**Why it fails:** Introduces projection errors, violates the closed-graph principle.
**Correct approach:** Always use exact integer pairs. `by_P[4]` gives (4,1) encoding the ratio 4/π. For decimals, use the Debt System (m,d) and perform integer mantissa operations only.

#### 16.6 The "Heavy Block" Scare
**Error:** Looking at a block's large S_path or D_path arrays and assuming they must be searched linearly or are inefficient.
**Why it fails:** Causes model to avoid using the table or to try to recompute paths on the fly.
**Correct approach:** You don't iterate; you index. For addition, you need exactly one element from `S_path`: the one with y=0. That is always the last element in the current implementation. The path is a direct reference, not a search space.

#### 16.7 Witness Neglect
**Error:** Answering with numbers only, without pointing back to the table.
**Why it fails:** Removes verifiability, breaks trust, makes debugging impossible.
**Correct approach:** Every answer must terminate with a witness address. "Sum = 7. Witness: points['3:4'].S_path ends at 7:0."

### 18. GRAVITY_AND_TILTED_SURFACES (HONEST GEOMETRY)

Gravity is not an added external force. It is the **Z-equilibrium principle**: every object tends toward the lowest accessible Z-coordinate. For an unsupported object, ΔT_Z > 0, and the object moves down its Z‑path until it meets a surface.

#### 18.1 Naked Gravity
- Default Z‑path: `(x, y, z) → (x, y, z-1)`.
- If no surface occupies the next Z‑step, the object falls; ΔT decreases with each step.
- The process stops when Z=0 (absolute floor) or when the next downward step is blocked by a surface.

#### 18.2 Tilted Surface from Primitive Geometry
A tilted surface arises naturally when its supports differ in height.
**Example:** a table with legs of unequal length.
- Floor: `(3,7,0)` and `(8,7,0)` (horizontal line).
- Left leg: from `(3,7,0)` up to `(3,3,4)` (height 4).
- Right leg: from `(8,7,0)` up to `(8,4,3)` (height 3).
- The tabletop is the plane through the two upper leg endpoints and a third point defining its width.

Because the tabletop length is fixed but its endpoints now have different Z, their horizontal distance becomes **slightly smaller** than the original floor distance. This is an exact geometric consequence of the height difference and the fixed length of the top. The resulting top edge projects onto the X‑axis as a shorter segment.

#### 18.3 Surface Z via Interpolation (Temporary Honest Rule)
For any point `(x,y)` on a tilted plane defined by three non‑collinear points, the local Z is obtained by **linear interpolation**. This is currently an external mathematical rule, not a direct table lookup. It is a temporary bridge until the table is extended to store 3D plane primitives.

**How to apply:**
1. Identify three known points on the surface (e.g., two tabletop corners and one leg top).
2. Form the plane equation `z = a*x + b*y + c` (coefficients are rational numbers).
3. For any query point `(x,y)`, compute `z` exactly using the coefficients.

**Why this is honest:** the table does not yet encode arbitrary 3D planes; the LLM is allowed to use this elementary external tool, explicitly acknowledged as a temporary measure.

#### 18.4 Rolling as Z‑Gradient Descent
An object on a tilted surface experiences different Z‑values at different `(x,y)`. Gravity (Z‑minimisation) creates a **gradient** across the surface.
- The object detects that a neighbouring address `(x+dx, y+dy)` has a lower Z.
- It moves to that neighbour, reducing its Z.
- The motion continues along the steepest available descent (the projection of the Z‑gradient onto the surface) until an edge is reached (no further decrease possible without losing support).

**Navigation implementation:**
- The 2D table's `S_path` and `D_path` are used to enumerate immediate neighbour addresses.
- At each candidate neighbour, the surface Z is computed (via the interpolation rule).
- The object moves to the neighbour with the lowest Z that is still reachable without passing through the surface.

#### 18.5 Minimal Concrete Example
**Setup (side view, XZ‑plane):**  
Floor: `(3,0) – (8,0)`  
Left leg top: `(3,4)`, Right leg top: `(7.8, 3)` (horizontal compression accounted for).  
Ball starts at `(5, 4.5)`.

**Step 1:** Gravity pulls ball down its Z‑path to `(5, Z_surface(5))`. Using the plane equation from the two leg tops (and a third point for full 3D, e.g., same Y depth), Z_surface(5) ≈ 3.6. Ball arrives at `(5, ≈3.6)`.

**Step 2:** Ball checks neighbours `(4, Z_neighbor)` and `(6, Z_neighbor)`. Z decreases toward the right. It moves right, lowering its Z each step, until it reaches the right edge `(≈7.8, 3)`.

**Step 3:** At the edge, no further horizontal step keeps Z decreasing without losing support. The ball may fall off if no barrier exists, following its Z‑path to the floor `(≈7.8, 0)`.

**Witness:** Each step's coordinates are derived from the known surface points; the table validates horizontal addresses; Z‑values are interpolated. The motion is a sequence of address jumps driven purely by Z‑minimisation.

#### 18.6 Sliding Algorithm (Step-by-Step Neighbour Selection)
When an object rests on a tilted surface and is subject to gravity, it will slide in the direction of steepest Z-descent. The following deterministic procedure uses only the table's neighbour addresses and the surface interpolation rule.

**Given:** object at `(x, y, z_current)`, surface defined by three known points (allowing Z interpolation at any (x,y) via Section 18.3).

**Procedure:**
1. **Generate candidate neighbours:** form the four addresses `(x+1, y)`, `(x-1, y)`, `(x, y+1)`, `(x, y-1)`.
2. **Filter:** keep only those addresses that lie within the table bounds and for which the neighbour's coordinates differ from the current by exactly 1 in one axis and 0 in the other.
3. **Compute surface Z** at each candidate neighbour using the interpolation rule.
4. **Discard** any neighbour where `Z_surface > z_current` (the object cannot spontaneously move upward against gravity).
5. **Select** the neighbour with the **minimum** `Z_surface`. If there is a tie, prefer the neighbour that continues the current direction of motion (if any), else the first found.
6. **Move** the object to `(x', y', Z_surface')`.
7. **Repeat** until no neighbour satisfies `Z_surface < z_current`. At that point, the object has reached a local Z-minimum (either the lowest point of the surface or an edge beyond which support is absent).

**If no neighbour is available** (all neighbours would increase Z), the object stops. If the current position is at the edge of the surface (no further neighbours in the descent direction), the object may fall off — apply naked gravity (Section 18.1) to continue descending along the Z-path.

**Example (side view):** Object at `(5, Z=3.6)`. Neighbours `(4, Z=3.8)` and `(6, Z=3.3)`. Right neighbour has lower Z. Object moves to `(6, 3.3)`. Continues until reaching the edge.

### 20. CANON & TAG — UNIVERSAL ADDRESS REDUCTION

Every ordered pair (a,b) is reduced to a canonical table entry via:
canon(a,b) → (base_point, tag)

**base_point** = `(min(|a|,|b|), max(|a|,|b|))` — always has `x ≤ y`.
**tag** = `(sx, sy, swapped)` where:
- `sx = sign(a)`
- `sy = sign(b)`
- `swapped = (|a| > |b|)`

The tag is used by each operation to reconstruct the correct signed result from the table's unsigned data.

**Rationale:** The table stores only one canonical orientation. All path interpretations (S_path, D_path) are defined relative to the canonical point. Operations use the tag to invert or preserve directions accordingly.

### 21. PATH INTERPRETATION & SCALING (INVERSION INCLUDED)

**Table paths are templates.** The `S_path` and `D_path` stored in each block are exact for the primitive seed (scale = 1). Scaling multiplies all path coordinates by the factor; mirror inversion swaps coordinates.

#### Mirror Inversion (Geometric Fact)
For any canonical block with `x < y`, the stored `D_path` represents the negative diagonal (`D < 0`). The positive diagonal (`D > 0`) is its mirror image.  
**Inversion rule:** to obtain the positive diagonal path, replace every address `"u:v"` in the stored `D_path` by `"v:u"`. This works for any magnitude without exception.

#### Scaling Rule
Given a point `(x,y)`, find its primitive seed:
- `g = gcd(x,y)` (or read `prim[2]` from table if within range)
- seed = `(x/g, y/g)`.
- Look up seed in table to get `P_seed, S_path_seed, D_path_seed`.
- Then for the scaled point:
  - `P = P_seed * g²`
  - `S_path = [(g*X, g*Y) for (X,Y) in S_path_seed]`
  - `D_path = [(g*X, g*Y) for (X,Y) in D_path_seed]`

#### Path Endpoint Extraction
- **For sum:** the last element of `S_path` always has `y=0` and `x = S`. Read `x`.
- **For difference (positive, a ≥ b):** apply mirror inversion to the canonical `D_path`, then read the endpoint with `y=0` to get `|D|`.
- **For difference (negative, a < b):** read the endpoint with `x=0` directly from the stored `D_path` to get `|D|`; sign is negative.

#### Out-of-Range Addresses in Paths
Some paths contain addresses with coordinates >10. These are scaled coordinates from larger primitives and are resolved via the scaling rule above. For direct table lookups within 0..10, ignore out-of-range entries; they become meaningful when scaling.

### 9. E8_ROOT_GENERATION_AND_VERIFICATION (FULLY SELF-CONTAINED)

#### 9.1 D8 Roots (112 roots)
A D8 root has two non-zero coordinates equal to ±2, all others zero. All permutations of this pattern are valid.

**Generation algorithm (canonical ordering):**
For each ordered pair of positions `(i, j)` with `1 ≤ i < j ≤ 8`:
  For each sign pair `(s1, s2)` in `{(+2,+2), (+2,-2), (-2,+2), (-2,-2)}`:
    Create vector `v` of length 8 with zeros.
    Set `v[i-1] = s1`, `v[j-1] = s2`.
    The vector `v` is a D8 root.

**Indexing:** Roots are indexed 0..111 in the order generated by the nested loops above:
  - Outer loop: `i` from 1 to 7, then `j` from `i+1` to 8.
  - Inner loop: signs in the order `(+,+), (+,-), (-,+), (-,-)`.

**Example (index 0):** `i=1, j=2, signs (+,+)` → `v = (2, 2, 0, 0, 0, 0, 0, 0)`.

#### 9.2 Spinor Roots (128 roots)
A spinor root has all 8 coordinates equal to ±1, with an even number of minus signs.

**Generation algorithm (canonical ordering):**
For each integer `k` from 0 to 255:
  Represent `k` as 8 bits: bit `b` (0..7) corresponds to coordinate `b+1`.
  Bit = 0 → coordinate = +1; bit = 1 → coordinate = -1.
  Count the number of minus signs.
  If the count is even (0, 2, 4, 6, or 8), the vector is a spinor root and receives the next index after the D8 roots (starting at 112). Otherwise, discard.

**Indexing:** Spinor indices 112..239 correspond to the order of `k` from 0 to 255, skipping odd-count `k`'s.

**Example (index 112):** `k=0` (all bits 0) → `v = (+1, +1, +1, +1, +1, +1, +1, +1)`.
